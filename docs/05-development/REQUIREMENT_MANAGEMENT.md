## 🎯 核心理念

> **敏捷开发 + 文档归档 = 可持续发展**

### 敏捷开发的文档原则

| 原则 | 说明 | 执行方式 |
|------|------|---------|
| **及时归档** | 开发完成后立即整理 | 每个任务完成后 30 分钟内 |
| **最小可行文档** | 只写必要的 | 需求 + 设计 + 变更 |
| **版本追踪** | 记录每次变更 | Git + CHANGELOG |
| **需求追溯** | 代码关联需求 | 需求 ID + Commit Message |

## 🔄 需求管理流程

### 需求生命周期

```
需求提出
   ↓
需求分析 (1-2 天)
   ↓
编写需求文档
   ↓
需求评审
   ↓
技术设计
   ↓
开发实现
   ↓
测试验证
   ↓
文档归档 ✅
   ↓
需求关闭
```

### 每日归档流程

```
每日 17:00-17:30 (开发收尾时间)
   ↓
1. 运行同步检查
   python3 projects/automation/scripts/sync_checker.py
   ↓
2. 整理今日代码
   ↓
3. 编写/更新需求文档
   ↓
4. 提交 Git (含文档)
   ↓
5. 更新 CHANGELOG
   ↓
6. 标记需求状态
```

## 📊 需求优先级定义

### P0 - 紧急重要

| 特征 | 说明 |
|------|------|
| **影响** | 核心功能缺失 |
| **时间** | 本周必须完成 |
| **资源** | 全力投入 |

### P1 - 重要不紧急

| 特征 | 说明 |
|------|------|
| **影响** | 重要功能改进 |
| **时间** | 本月完成 |
| **资源** | 正常排期 |

### P2 - 不重要

| 特征 | 说明 |
|------|------|
| **影响** | 锦上添花 |
| **时间** | 有空再做 |
| **资源** | 剩余资源 |

## 🎯 最佳实践

### ✅ 应该做的

1. **需求先行**: 开发前先写需求文档
2. **小步快跑**: 需求拆分成小任务
3. **及时归档**: 完成后立即整理
4. **版本追踪**: Git + CHANGELOG
5. **需求追溯**: 代码关联需求 ID

### ❌ 不应该做的

1. **边做边改**: 没有需求文档就开发
2. **事后补档**: 开发完很久才补文档
3. **口头需求**: 没有书面记录
4. **需求蔓延**: 不加控制地增加需求
5. **文档分离**: 代码和文档不同步
